Rubric

Grade components

Your lab grade is a combination of correctness and program quality. The correctness is measured by tests and manual inspection by the instructor. Program quality is evaluated by a manual review of the code. Quality is assigned a score between 0 and 10. The rubric below outlines factors that contribute to that score.

Criteria used to assess program quality

Programs should adhere to good program style. There are many resources online that describe standard practices, such as Google’s style guide. Such guides may prove to be a useful reference.

In addition to evaluating basic program style, your submission will be evaluated by these factors:

  1. Program design: code makes appropriate use of helper functions to solve natural sub-problems; arguments to method are appropriate; programmer follows the DRY (Don’t Repeat Yourself) principle;
  2. Readability: the code has a logical flow that is easy for an experienced programmer to follow. Print statements that were added for debugging purposes should be removed.
  3. Efficiency: at a minimum, code avoids computation that is strictly unnecessary (e.g., having two for loops that could easily be combined into one). Some assignments may specify additional criteria around efficiency.

  4. Effective use of comments: Javadocs (if not already provided) can be used to document the purpose of a method. Comments within the method body should be used sparingly, to highlight the role of certain key steps. Avoid commenting every line and avoid comments that say in words what is obvious from the code. Here is an example of an unnecessary comment.

     x = x + 1;   // increment x by one Avoid writing comments like that.
  5. Commented-out code: comments should be used for comments only. Do not leave commented out code in your final submission.
  6. Names (for Variables, Methods, Classes): names for variables should be meaningful and concise. Some very short names are okay if used in the appropriate context. Examples: variables i,j, and k are conventionally used as integer indexes; variables like p,q,r make sense in the context of propositional logic; variables like S, T are appropriate as identifying arbitrary sets. However, avoid using arbitrary short names – such as a, b, c, d – and avoid having similar names refer to objects that have different types (e.g., using var1 to refer to a set and using var2 to refer to an element of the set). Method names should accurately capture the methods purpose.
  7. Indentation and whitespace: Blocks of code should be clearly delineated using consistent indentation. Most editors have the ability to automatically format your code; there are also online tools. Use appropriate white space between logical code blocks. Use appropriate white space within lines of code. Binary operators — e.g., “=”, “+”,— should generally have a space on either side.
  8. Extra files uploaded: Please upload only the .java files and only the files that contain code that you wrote (i.e., if I provide a file that you never modify, you don’t need to send it back to me!).
